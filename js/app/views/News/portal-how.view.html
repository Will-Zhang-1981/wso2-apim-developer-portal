När vi byggde den nya portalen tog vi fasta på vad utvecklarportalen handlar om, dvs API:er. Därför valde vi att först designa all backend-funktionalitet som API:er, och att dokumentera dessa med hjälp av Swagger Därefter använde vi dessa specifikationer för att generera delar av koden som blev vår backend och de API:er som klienten kommunicerar med för att läsa och skriva data. Klienten kunde därmed implementeras som en ren web-applikation med HTML, CSS och JavaScript. Det betyder att vi har byggt vår nya portal enligt nedan:

  • WSO2 API Manager. Vår API-plattform bygger på 100% öppen källkod i Java och kommer från företaget WSO2. För att bygga vår nya portal har vi därför utgått ifrån den funktionalitet som finns i produkten "out-of-the-box", vilket innebär att vi har utnyttjat de web-tjänster som finns inbyggda i denna produkt. Ovanpå detta lager av tjänster, som är såväl SOAP- som REST-baserade, har vi byggt våra egna REST-baserade API:er, som består av 2 delar; ett API hanterar allt som har med utvecklarkonton att göra, och det andra hanterar resten.
  • Jersey RESTful Web Services framework. För att bygga våra API:er enligt ovan har vi använt ramverket Jersey.
  • HTML/CSS. Vi har använt Bootstrap 3 och försökt använda standardkomponenter ur detta ramverk för att skapa ett intuitivt och responsivt gränssnitt.
  • AngularJS. Med hjälp av AngularJS har vi kunnat bygga en användarupplevelse som lämpar sig väl för interaktion mellan användare, klient och vårt backend API.

En fördel med att implementera all funktionalitet som API:er är att vi, om intresse finns, kan välja att publicera dessa API:er via vår API Gateway. Det skulle innebära att ni utvecklare kan prenumerera på dessa API:er och utveckla egna appar som gör samma sak som vår egen utvecklarportal, men som t.ex. är särskilt anpassade för mobila enheter. Om detta låter intressant, hör av dig till oss på Västtrafik så tar vi med dina synpunkter när vi fattar beslut om huruvida vi ska göra detta eller inte.
